# 洁净区暖通计算工作台：Agent 开发执行计划

> 本文是可以直接交给其他开发 Agent 执行的主任务书。执行前必须先阅读本文列出的权威资料，不得自行改变已确认的产品边界和计算语义。

## 0. 权威资料与优先级

按以下顺序解决冲突：

1. `暖通空调计算表模板（简化）.xlsx`：参数含义、单位、默认值、有效公式和显示格式的计算权威。
2. `docs/requirements.md`：产品范围、交互、数据交换、部署和验收要求。
3. `CONTEXT.md`：领域术语和对象关系。
4. `docs/adr/*.md`：不可随意推翻的架构决策。
5. 本文：实施顺序、代码边界、任务包和交付证据。

若 Excel 中出现断链公式、缺失关系或与需求文档矛盾，不得猜测。记录到 `docs/catalog/pending-formulas.md`，等待业务确认。

## 1. 最终目标

交付一个可在 Linux 或 Windows 内网服务器原生部署的轻量 Web 应用：

- 单一共享入口账户，不建设个人用户和权限体系。
- 项目库长期保存项目、房间和方案版本。
- 当前 Excel 的有效公式等价迁移，普通页面不展示或允许编辑公式。
- 用户按房间录入参数，支持复制新建和 Excel 批量收资。
- 汇总页展示项目总需求和楼层需求。
- 计算总表保持“参数为行、房间为列”，最多支持 200 个房间。
- 结果 Excel 提供汇总和全部房间参数，供外部机构进行设备选型。

## 2. 不可变约束

- 技术栈：Python Flask 单体、Jinja、HTMX、少量原生 JavaScript、SQLite、`openpyxl`。
- 不使用 Docker、微服务、独立 React 工程或独立数据库服务。
- 只支持桌面最新版 Edge/Chrome，目标宽度不小于 1366px。
- 方案版本默认可编辑；需要保留原版本时使用 Fork。
- 每个方案版本绑定一个不可变计算规则版本。
- 同一方案版本只允许一个编辑会话；其他会话只读或 Fork。
- 用户明确点击“保存方案”才持久化；不得自动覆盖方案版本。
- 项目删除和版本删除均为永久删除，不提供回收站和自动备份。
- 收资文件全量覆盖同编号房间，空白也清空；任一错误导致整批不导入。
- V1 不包含设备型号推荐、阈值报警、PDF、移动端、审批、实时协作和公式技术保密。

## 4. 核心领域模型

### Project

- `id`
- `project_code`：全项目库唯一
- `name`
- `client`、`address`、`remark`：可空
- 创建时间、更新时间

### SchemeVersion

- 隶属于 Project
- `version_no`：项目内 `V1、V2…`
- `name`、`remark`
- `source_version_id`：Fork 来源，可空
- `rule_version`
- `city_code`
- 标准气象值、覆盖值和覆盖标记
- 当前保存状态及保存时间

### Room

- 隶属于 SchemeVersion，而不是直接共享于 Project
- `room_code`：方案版本内唯一
- `name`、`floor`、`area_group`
- `inputs_json`：按稳定参数编码保存全部可编辑输入
- `results_json`：按稳定参数编码保存计算结果及状态

### CalculationRuleVersion

- 不需要让用户编辑。
- 由代码和静态数据共同定义，发布后不可变。
- 规则版本必须包含气象参数库版本和参数目录版本。

### EditLock

- `scheme_version_id`
- 浏览器会话标识
- 最后活动时间
- 30 分钟无活动自动失效

## 5. 稳定参数编码

所有参数必须具有不依赖 Excel 单元格位置的稳定编码，例如：

```text
room.area_m2
room.ceiling_height_m
indoor.summer.temperature_c
load.summer.total_cooling_kw
airflow.supply.design_cmh
ffu.required_count
dcc.water_flow_m3h
```

参数目录至少包含：

- 编码、中文名、单位、数据类型、显示精度。
- 所属分组和顺序。
- 输入/默认值/计算结果属性。
- 必填性、可选值、默认值。
- Excel 工作表及单元格来源。
- 是否可累加及汇总方式。
- 是否出现在收资文件、总表、汇总页和结果文件。

页面、Excel 和计算内核均引用参数编码，不复制名称和单位常量。

## 6. 计算内核契约

计算内核必须是无数据库、无 Flask 上下文的纯函数模块。

建议接口：

```python
@dataclass(frozen=True)
class CalculationContext:
    rule_version: str
    outdoor: dict[str, Decimal]
    room_inputs: dict[str, object]

@dataclass(frozen=True)
class ParameterResult:
    status: Literal["value", "pending_formula", "not_applicable", "error"]
    value: Decimal | str | None
    message: str | None = None

def calculate_room(context: CalculationContext) -> dict[str, ParameterResult]:
    ...
```

要求：

- 数值计算优先使用 `Decimal` 或经过验证的等价精度策略。
- 只有 Excel 明确使用 `ROUNDUP`、`FLOOR` 等时才改变实际值。
- 显示精度由参数目录负责，不得反向污染中间结果。
- 空输入不得产生 `#DIV/0!`；应返回可解释状态。
- 开发期断链公式返回 `pending_formula`，V1 上线前必须清零。
- 每个公式测试都要指出 Excel 来源单元格和对照输入。

## 7. 页面与操作契约

### 登录页

- 一个共享账户和口令，来自服务器配置。
- 登录失败显示统一错误，不暴露配置细节。

### 项目库

- 搜索项目编号/名称。
- 新建项目或从收资文件新建。
- 显示项目、城市、版本数、最近更新时间。
- 永久删除前明确告知会删除全部房间和方案版本。

### 项目工作台

- 顶部固定显示项目、当前版本、规则版本和未保存状态。
- 版本支持读取、保存、Fork、删除。
- 编辑锁冲突时只读，并允许 Fork。
- 房间侧栏“应用”只更新工作副本并重算；顶部“保存方案”才写入数据库。

### 汇总展示

- 项目总需求、楼层需求、关键数量表。
- 面积、冷热负荷、风量、设备数量、功率和水量等可累加指标求和。
- 温湿度、洁净等级、压差、换气次数、风速和比例不求和，只显示范围或分布。

### 计算总表

- 参数纵向、房间横向。
- 左侧参数列固定；房间列横向滚动或虚拟渲染。
- 支持楼层、区域筛选和参数搜索。
- 支持分组折叠：基础参数、洁净与正压、冷热负荷、风量、FFU、干盘管、回风夹道与百叶。
- 表格只读；选择房间打开侧边编辑面板。

### 房间侧栏

- 分为基本信息、室内条件、洁净与排风、负荷、设备与构造参数。
- 只展示可编辑参数，不显示公式。
- 支持复制新建；复制全部可编辑参数及楼层/区域，不复制编号、名称和结果。

## 8. Excel 契约

### 收资文件

- `.xlsx`，至少包含填写说明、项目信息与室外条件、房间参数。
- 房间参数页保持参数为行、房间为列。
- 包含参数名、单位、必填说明和可选值，不包含公式。
- 支持导出全部房间或选中房间。
- 向已有项目导入时只更新房间，不覆盖项目级信息。
- 从文件新建项目时读取项目级信息；项目编号重复则阻止。
- 同编号房间执行全量覆盖，空白清空原值。
- 导入预览显示新增、覆盖和冲突。
- 任一错误整批回滚，错误必须定位到工作表、参数行和房间列。

### 结果文件

- `.xlsx`，只导出值和单位，不导出公式。
- 工作表一：项目/楼层需求汇总。
- 工作表二：全部房间计算总表。
- 包含项目、方案版本、规则版本、导出时间和气象覆盖说明。
- 不支持反向导入。

## 9. Agent 任务包

### A01：字段、公式和气象数据目录

依赖：无。

工作：

1. 解析权威 Excel 的全部工作表、有效单元格、样式和公式。
2. 生成 `docs/catalog/parameters.md` 和机器可读 `data/parameter_catalog/parameters.json`。
3. 生成 `docs/catalog/formulas.md`，记录公式、依赖、单位、取整和来源单元格。
4. 生成 `docs/catalog/pending-formulas.md`。
5. 生成标准化气象数据文件，并验证城市条目数和关键字段。

验收证据：

- 参数目录覆盖表三全部参数。
- 每个有效公式有唯一输出参数和来源。
- 断链项无遗漏、无猜测。
- 自动校验脚本通过。

### A02：应用骨架和数据层

依赖：可与 A01 并行，但不得自行定义参数编码。

工作：

1. 建立 Flask 工厂、配置、共享入口验证和 SQLite 初始化。
2. 实现 Project、SchemeVersion、Room、EditLock 表和迁移。
3. 实现项目和方案版本服务。
4. 实现 30 分钟编辑锁及会话心跳。

验收证据：

- 单元测试覆盖唯一约束、Fork、永久删除和锁超时。
- 重启服务后数据可恢复。
- 并发保存不会静默覆盖。

### A03：规则版本与计算内核

依赖：A01。

工作：

1. 实现计算契约和 `rules_v1`。
2. 实现只读气象库、项目覆盖和恢复标准值。
3. 建立 Excel 黄金样例和公式逐项对照测试。
4. 完成全部待补充公式闭环。

验收证据：

- 所有有效公式逐项对照通过。
- 空房间和缺失可选值不产生 Excel 错误码。
- V1 不存在 `pending_formula`。

### A04：收资与结果 Excel

依赖：A01、A02、A03。

工作：

1. 实现收资模板生成和选中房间导出。
2. 实现新项目导入、已有项目全量覆盖导入和预览。
3. 用 SQLite 事务保证整批成功或整批失败。
4. 实现结果双表导出。

验收证据：

- 新增、覆盖、冲突、空白清除、重复编号、非法值均有测试。
- 失败导入前后数据库快照一致。
- 导出的汇总和总表与网页当前方案一致且不含公式。

### A05：项目、版本和编辑流程 UI

依赖：A02；可使用模拟计算结果先开发。

工作：

1. 登录、项目库、项目新建/删除。
2. 版本读取、保存、Fork、删除和未保存提示。
3. 编辑锁只读状态和会话心跳。
4. 房间新建、复制、编辑和删除。

验收证据：

- Edge/Chrome 桌面流程截图或录屏。
- 离开未保存页面有提示。
- 锁冲突状态下无法修改原方案但可以 Fork。

### A06：汇总页、计算总表和房间侧栏

依赖：A01、A03；可用静态 fixture 先推进。

工作：

1. 实现汇总指标和楼层分组。
2. 实现总表冻结列、房间横向虚拟渲染、分组折叠和筛选。
3. 实现房间侧栏分组输入和应用重算。
4. 验证 200 房间项目。

验收证据：

- 1366px 与 1440px 视口无关键文字截断。
- 200 房间可筛选、滚动、编辑和重算。
- 可累加和非累加指标行为符合目录定义。

### A07：集成、部署和交付

依赖：A01—A06。

工作：

1. 端到端集成全部页面和服务。
2. 提供 Linux `systemd` 与 Windows 服务安装、升级、卸载说明。
3. 提供配置示例、数据目录约定和数据库迁移流程。
4. 完成验收矩阵、发行说明和已知限制。

验收证据：

- Linux 与 Windows 各完成一次干净环境部署演练。
- Edge 与 Chrome 完成完整主流程。
- 200 房间项目完成导入、计算、保存、Fork、汇总和导出。

## 10. 推荐执行顺序与并行关系

```text
A01 字段/公式目录 ──┬──> A03 计算内核 ──┬──> A04 Excel
                    │                   └──> A06 总表/汇总
A02 应用骨架 ───────┼──> A05 项目/版本 UI
                    └──────────────────────> A07 集成部署
```

- A01 是最重要的前置任务，未完成前不得大规模实现业务页面或公式。
- A02 可与 A01 并行，但只能使用临时参数 fixture。
- A05 和 A06 可以并行，必须共用同一参数目录和视觉组件。
- A07 只能在所有任务包的验收证据齐全后开始。

## 11. 测试与完成门槛

最低测试层级：

- 参数目录结构校验。
- 每个公式的单元测试和 Excel 黄金对照测试。
- 项目、版本、Fork、删除和锁的服务测试。
- 导入事务、覆盖和错误定位测试。
- 汇总规则测试。
- 主流程浏览器测试。
- Linux/Windows 部署冒烟测试。

完成 V1 前必须证明：

1. 表三全部参数已映射。
2. 所有有效公式均通过 Excel 对照。
3. 待补充公式数量为零。
4. 200 房间主流程可用。
5. 收资导入失败不会产生部分更新。
6. 历史方案规则版本不会被新规则改写。
7. 页面不展示或允许编辑公式。
8. 结果文件不包含公式。
9. Linux/Windows 均有可重复部署步骤。
